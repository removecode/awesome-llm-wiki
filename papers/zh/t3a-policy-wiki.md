## 摘要

本文研究：对于具有天然表格比较结构的跨机构政策冲突查询，是否可以通过预先将文档语料编译为结构化
Wiki 来提升效果。
我们定义了一项任务（T3a）：系统必须针对一个焦点机构和一个政策维度，识别焦点机构的规则，以及与其
不一致的其他机构，并提供文章级引用作为支撑。
我们基于来自 19 所机构的 22 份中国高校科研经费政策文件，构建了一个包含 28 个查询的基准。
黄金引用由一个基于 LLM 的流水线自动抽取，并配有可机械审计的后置检查；随后又在 6 个代表性查询上
进行了人工审计（审计 172 条黄金引用，其中 93.0% 被人工验证为正确或部分支持；见第 4.2.1 节）。
我们比较了三个共享同一语言模型生成器的系统：无检索系统、BM25 检索增强生成系统，以及一个基于
Wiki 的检索系统，其页面由同一语料离线编译而成。
Wiki 系统的引用召回率达到 0.242 ± 0.21，而 BM25 为 0.095 ± 0.12，无检索为 0.023 ± 0.03；
所有成对差异均具有统计显著性（p < 10⁻³，配对 Wilcoxon 检验）。
关键在于，Wiki 系统是在仅消耗约 BM25 输入上下文 token 数 29% 的情况下达到这一结果的，说明优势
并非由更大的上下文体量驱动。
失效模式分析显示，幻觉引用在三个系统之间单调下降（75% → 54% → 39%）。
我们报告了编译 Wiki 中两个由 LLM 引入、但引用级验证无法捕捉的失效模式。
我们将这一结果表述为关于跨文档比较查询中任务与表示之间结构对齐的设计启示——即在这一任务族上
“先编译、后查询”安排的概念验证——而不是相对于现代检索系统的一般性优越主张。

**关键词：** 检索增强生成；知识编译；引用忠实性；政策文件；大语言模型；中文 NLP；BM25；基准；
跨文档比较

## 1. 引言

语言模型的许多真实应用都需要综合多份描述同一类对象的文档中的信息：一组合同、一批监管备案文件，
或者若干高校的科研经费管理政策。
当用户询问某个机构的规则与其他机构有何不同时，基于检索增强生成（RAG）[1], [2], [28] 的语言模型
每次都必须隐式执行跨文档推理：检索相关条款、对齐其含义、识别差异，并给出引用。
这种推理不会被持久化；下一个问题会重新开始同样的工作。

Karpathy 最近提出了一种替代安排 [6]：将语料一次性离线编译成一个持久的结构化 Wiki，即一组 Markdown
页面，其中包含已经解析好的交叉引用和显式比较表；推理时再查询这个 Wiki。
该 gist 描述了这一模式，但没有对其进行评估。
本文在一个受控任务上检验该模式，并将结果表述为关于任务—表示对齐的设计启示，而不是对检索式方法
的一般性优越主张。

我们定义了一项跨机构政策冲突查询任务，称为 T3a：给定一个焦点机构和一个政策维度，识别该焦点机构
在该维度上的规则、规则不同的其他机构，以及支撑这些判断的文章级引用。
该任务天然具有表格性：最直接的人类答案是一张比较表，每个机构一行。
我们利用这一结构属性，并在第 5 节将其明确讨论为适用范围条件。
我们基于来自 19 所机构的 22 份中国高校科研经费管理政策文件构建了包含 28 个查询的基准。
我们评估三个匹配系统：没有检索的参数化基线、使用适合中文文本的字符 n-gram 分词器的 BM25 RAG
基线，以及检索预编译比较页面的 Wiki RAG 系统；三者均使用同一底层语言模型（glm-4-plus）。
我们在 (doc_id, article_number) 层级上，将系统输出与 LLM 抽取的黄金引用进行比较。
这是一个可机械审计的指标，不依赖于匹配 LLM 的自然语言复述，也不依赖于 LLM-as-judge 协议 [21]；
而后者在长文本事实性评估中的可靠性本身仍是一个开放问题。

在 28 个查询的基准上，Wiki RAG 系统的平均引用召回率达到 0.242，而 BM25 RAG 为 0.095，无检索
下限为 0.023。
在配对 Wilcoxon 符号秩检验下，三组成对差异均达到 p < 10⁻³，且 bootstrap 95% 置信区间均排除 0。
三项次级分析澄清了优势来源。
第一，纯检索测量显示，Wiki 页面包含的黄金引用数量甚至超过 k = 20 时的 BM25，因此优势并非检索
广度问题，而是结构化预编译问题。
第二，上下文预算分析显示，Wiki RAG 仅使用 BM25 RAG 约 29% 的提示输入 token 即取得这些结果，
排除了 Wiki 只是接收更多信息这一替代解释。
第三，对全部 84 个预测的失效模式分类显示，幻觉引用随检索升级单调下降（75% → 54% → 39%）。
为回应有关 LLM 抽取黄金标准有效性的担忧，我们对 6 个代表性查询（每个政策维度一个）进行了人工审计，
并报告一致率。

本文贡献如下：

- 定义了 T3a 任务，并在中国高校科研经费管理政策上构建了一个 28 查询基准，随同发布文章级黄金引用
  和经人工审计的子基准；
- 实现了带引用检查后置条件的 LLM-Wiki 编译模式，并与匹配的 BM25 RAG 基线进行并列比较，其中
  LLM 生成器相同，输入上下文预算经过量化；
- 提供了统计证据，并以纯检索、上下文预算和失效模式分析加以支撑，表明在本基准中特定的跨机构
  政策冲突查询任务族上，预编译 Wiki 页面优于扁平检索；
- 明确报告了编译 Wiki 中两个由 LLM 引入、但引用级验证无法捕捉的失效模式：一个发生在展示名称层面，
  一个发生在焦点归因层面；这些案例说明未来需要语义引用验证。

本文其余部分组织如下。
第 2 节回顾检索增强生成、知识编译和引用忠实性相关工作。
第 3 节形式化 T3a 任务，并描述 Wiki 编译流水线、三个基线系统和评估指标。
第 4 节报告实验结果，包括语料统计、基准黄金标准人工审计、上下文预算比较、主要指标、纯检索分析、
失效模式分解和详细案例研究。
第 5 节讨论为什么结构化 Wiki 在这一任务族上优于扁平检索、结果的适用范围条件，以及有效性威胁。
第 6 节总结全文。

## 2. 相关工作

本文工作的技术基础是 transformer 架构 [16]，以及建立在其上的大语言模型家族，包括 BERT [10]、
T5 [17]、GPT-3 [18]、GPT-4 [19] 和 LLaMA-2 [20]。
三条既有研究脉络与本文更直接相关。

### 2.1 检索增强生成

检索增强生成 [1] 为语言模型扩展了一个在推理时访问的非参数记忆，从而应对参数化知识存储的内在限制。
Gao 等人 [2] 对由此产生的检索器、融合策略和生成技术生态进行了全面综述；Sharma 和 Bhattarai [28]
则从架构、研究挑战和新兴前沿角度给出了互补综述，讨论了检索设计权衡、幻觉行为、模块化 RAG 架构，
以及现代 RAG 系统中扎根性评估的挑战。
BM25 [3] 仍然是衡量每一种新检索技术的经典稀疏检索基线，因此本文采用它。
本文所针对的查询时 RAG 局限在该文献中已广为人知：语言模型每次被提问时，都必须重新推导跨文档
结构，而没有任何持久化工件来编码它先前已经推断出的内容。

### 2.2 基于 LLM 的知识编译

2026 年 4 月，Karpathy [6] 发布了一份想法文件，描述了他在个人实践中发现有用的一种模式：与其在
查询时从原始文档检索，不如使用语言模型从来源中逐步编译出一个结构化、相互链接的 Markdown Wiki，
然后查询该 Wiki。
Wiki 是一个持久工件；交叉引用和矛盾在编译时解析一次，不需要在每次查询时重新发现。
该 gist 本身是一项提案，而不是实证研究；它提出了本文要回答的问题，即这种“先编译、后查询”的安排
是否会在可测量任务上优于匹配的查询时 RAG。
据我们所知，本文是首次在受控基准上对这一模式进行定量评估。

### 2.3 引用忠实性与长上下文推理

另一条并行研究关注 LLM 输出中引用的可靠性。
Menick 等人 [5] 使用基于人类偏好的强化学习，训练模型引用具体支撑性引文，并认为显式引用是可信
LLM 输出的基本构件。
Liu 等人 [4] 表明，语言模型并不能稳健地利用分布在长输入上下文中的信息：即便是显式支持长上下文的
模型，当相关信息位于长提示中部时，性能也会显著下降。
在这些方向基础上，近期工作扩展了引用感知生成的空间：ALCE [12] 引入了一个自动基准，用于从流畅性、
正确性和引用质量评估 LLM 引用；Self-RAG [22] 添加反思 token 来决定何时检索；In-Context RAG [24]
将检索纯粹表述为提示增强而无需再训练；Graph RAG [23] 将文档组织为图结构摘要，用于面向查询的推理。
我们的基于引用的评估建立在第一条脉络上；第 5.1 节提出的机制性解释则连接到第二条脉络，因为编译 Wiki
可以被理解为将相关信息从长原始文档的中部移出，放入聚焦的比较表中，在那里相邻性使推理局部化。

### 2.4 中文政策文本 NLP

我们尚未发现此前有基准以本文使用的文章级引用粒度，针对中文跨机构政策冲突查询。
本文语料——来自 19 所中国高校的 22 份科研经费管理政策文件——专为这一目的从公开官方来源汇编而成，
并随评估档案发布。
本文工作与 LEGAL-BERT [14] 等领域专用预训练模型以及 BEIR [13] 等更广泛的 IR 基准互补：这些方法
训练或评估通用检索器和编码器，而本文固定生成器，并对比具有不同结构承诺的检索基底。

## 3. 方法

### 3.1 问题设定

我们定义一项跨机构政策冲突查询任务，并称其为 T3a。
输入是一个政策文档语料 D = {d₁, d₂, ..., dₙ}，每个机构有一份或多份文档，并划分为有限机构集合 I。
每份文档 dᵢ 被分割为条款 aᵢ,ₙ，其索引为条款编号 n，遵循中国行政文书中通行的“第 N 条”编号方式。
我们固定一组较小的封闭政策维度 Δ（在本基准中有六个：间接费用、劳务费、设备、金额阈值、比例阈值、
以及横向与纵向经费）。

一个 T3a 查询是一个元组 q = (i*, δ)，其中 i* ∈ I 是焦点机构，δ ∈ Δ 是政策维度。
期望回答包含三个组成部分：
（a）关于焦点机构 i* 在维度 δ 上规则的自然语言陈述；
（b）规则与 i* 不同的其他机构集合 F；
（c）支撑性引用集合 C，其中每个引用 (i, n) 指向机构 i 的政策文档第 n 条。

在评估中，我们使用元组级 precision、recall 和 F1，将预测引用集合 C 与黄金引用集合 C* 比较。
我们将引用召回率指定为主要指标；第 3.4 节解释为什么替代的基于冲突的指标在本基准上不适合显著性检验。

### 3.2 Wiki 编译流水线

我们的 Wiki 编译流水线将原始政策语料转换为一小组比较页面，每个政策维度 δ 一个页面，每个页面包含
一张按机构逐行排列、并附带文章级引用的表。
图 1 展示了这一编译步骤位于原始语料与查询时检索基底之间的位置。
算法 1 给出了按维度执行的循环。
下面描述六个组成部分。

![图 1](/src/image/t3a-policy-wiki/fig1.png)

**图 1. 系统架构。** 原始政策语料被处理两次：
（i）进入 BM25 索引，供查询时的 `bm25_rag` 基线使用；
（ii）经过 Wiki 编译器，由 LLM 生成按政策维度划分的比较表页面，并以 Markdown 存储，供查询时
`wiki_rag` 使用。
`no_retrieval` 基线只接收机构列表，不接收来源文本。
三个系统共享同一 LLM 生成器（glm-4-plus）。

**条款分割与手册排除。** 每份文档通过定位规范的“第 N 条”标题模式，在条款边界处切分。
分割器同时处理阿拉伯数字和中文数字形式的 N，并生成文档内部字典 aᵢ,₁, aᵢ,₂, ...。
语料中有一份文档采用问答手册格式，而非编号条款格式；其结构与文章级引用不兼容，因此被排除在
BM25 索引和 Wiki 编译之外。
这一排除由机构注册表中的 `doc_format` 字段配置，最终保留 683/685 条进入索引。

**证据窗口。** 对于给定维度 δ 和文档 dᵢ，证据窗口基于关键词匹配，选择 dᵢ 中最多八条与 δ 最相关的
条款。
每个维度带有一个小型人工整理关键词列表；对 `indirect_cost`，我们使用的关键词包括“间接费”、“管理费”
和“提取”。
这些关键词列表是在语料探索期间迭代开发的，目标是百分之百覆盖每个维度的规范中文术语。
证据窗口是按维度、按文档应用的过滤器：它将 LLM 必须阅读的输入从数百条缩减到十几条或更少的相关条款。
这既降低成本，更重要的是减少编译时 LLM 的上下文负担，从而缓解 Liu 等人 [4] 所记录的长上下文退化。

**按维度的比较表提示。** 对每个维度，我们维护一个以 YAML 定义的提示，要求 LLM 以固定 schema 输出
Markdown 比较表：机构、规则、具体数值、来源引用。
来源引用必须使用 `[doc_id:art.N]` 语法，指向证据窗口中包含的条款。
提示明确要求模型将未规定该维度的机构标为沉默，而不是编造规则。
编译时温度设为 0.2，以允许从词汇失误中进行小范围恢复，同时保持表结构稳定。
完整提示模板、证据窗口配置和 YAML schema 随代码档案发布（见数据可用性声明）。

**引用检查器。** 生成页面会被解析出 `[doc:art.N]` token。
对每个 token，我们验证：
（i）`doc_id` 是注册表中已知机构；
（ii）条款编号 N 在该文档的有效范围内。
该检查刻意保持保守：它捕捉两类最常见幻觉（未知机构 id、超出范围的条款编号），但不验证条款内容
是否匹配关于它的声明。
暴露此类语义错误需要另一种机制；第 4.7 节讨论当前检查的一项失效，第 5.2 节讨论语义引用验证的
具体方向。

**B5 防御。** 如果引用检查器标记错误，页面会与结构化错误列表一并返回给 LLM，请其输出修正版本。
我们允许最多 r_max = 2 次重试，并只缓存最终通过的版本。
在我们的语料上，该循环最多一次重试即可对 100% 的页面收敛，最终引用检查器通过率为 1060/1073 = 98.8%。
13 个残余失败是在两个机构中出现的未知条款错误，这些机构的真实条款数接近 LLM 选择的外推编号；
防御循环没有移除它们，因为不等式两侧看起来都合理。
我们保留这些错误，并在失效模式分析中记录。

**SQLite 结果缓存。** 每次 LLM 调用都以元组 (model, messages, temperature) 为键，并将响应持久化到
本地 SQLite 文件。
缓存策略很保守：如果任何后置检查拒绝了响应，则不写入该响应。
在典型的实验重跑中，命中率超过 90%，使迭代评估能以边际成本完成。

该流水线的产物是一个 Markdown 文件目录，每个政策维度一个文件；这些文件共包含 19 个机构、6 个维度上
的 1,073 条文章引用，是一个紧凑、可审计的基底，可供人类读者查询，也可在我们的实验中供下游 LLM
作为问答代理使用。

#### 3.2.1 算法：Wiki 编译

**算法 1. 一个政策维度 δ 的 Wiki 编译**

```text
INPUT: corpus D, dimension δ, LLM M, max retries r_max
OUTPUT: comparison page for δ (Markdown)
Step 1: E_δ ← EvidenceWindow(D, δ)
Step 2: page ← M(prompt_δ, E_δ)
Step 3: for r = 1 to r_max:
          (ok, errs) ← CheckCitations(page, D)
          if ok: return page
          page ← M(prompt_δ⁺, E_δ, errs)         // B5 defense
Step 4: return page, flag unresolved
```

### 3.3 基线系统

我们比较三个系统。
三者共享同一底层语言模型（智谱 AI 家族的 glm-4-plus 模型）、同一解析与评分代码，以及同一组基准查询。
它们唯一的差异是查询时注入 LLM 提示的上下文。
这一匹配设计将检索基底的影响与生成器的影响隔离开来。
第 4.3.1 节量化了由此产生的上下文预算，并显示 Wiki 系统实际上使用的输入 token 明显少于 BM25 系统。

**`no_retrieval`。** LLM 接收查询以及 19 个机构的封闭列表（使用 21 个机构标识符，其中多文档机构带有
版本后缀），但不接收任何来源文本。
该基线的目的不是竞争，而是建立下限：它衡量生成器在没有任何文档扎根时，仅凭参数化知识能够生成什么。
我们提供机构列表（而不是要求 LLM 从预训练中回忆机构名称），使三个系统共享同一标识符空间；
这使预测冲突集合可以在系统之间直接比较。

**`bm25_rag`。** 除查询外，LLM 还接收来自政策语料 BM25 索引的 top-k 条款 [3]。
我们的 BM25 实现使用字符级 n-gram 分词器（n ∈ {2, 3}），这适合词边界不显式的中文文本，也与中文
预训练语言模型既有工作一致 [15]；全程使用标准参数 k₁ = 1.5 和 b = 0.75。
索引建立在保留文档的 683 条条款上，手册文档被排除。
主要结果中使用默认值 k = 10；第 4.5 节考察 k 的影响。
我们选择 BM25 而不是 DPR [7]、ColBERT [8] 或 BGE-M3 [9] 等稠密检索器，是因为 BM25 无需微调，
并提供公平的稀疏检索基线；实证稠密检索比较在第 6 节被列为未来工作。

**`wiki_rag`。** 除查询外，LLM 还接收第 3.2 节编译流水线生成的、与查询维度 δ 对应的整页政策维度
Wiki 页面。
不再执行进一步检索：该页面是一个离线写入一次的单一预编译工件。
这是 Karpathy [6] 模式到定量评估系统中的直接转换。

三个系统均被要求以同一格式输出答案：一段简短的自由文本规则陈述，加上一个 JSON 块，其中包含
`value` 字段和 `conflicts` 机构标识符列表。
引用以内联方式出现在自由文本部分，采用 `[doc:art.N]` 语法。

### 3.4 评估指标

给定预测引用集合 C 和黄金集合 C*，我们以标准方式计算元组级引用 precision、recall 和 F1，定义见
式 (1)–(3)：

```text
precision = |C ∩ C*| / |C|,                     (1)
recall = |C ∩ C*| / |C*|,                        (2)
F1 = 2 · precision · recall / (precision + recall).   (3)
```

引用在 (doc_id, article_number) 粒度上匹配；匹配机构但不匹配条款不授予部分分。

**主要指标：引用召回率。** 我们的假设是，`wiki_rag` 比扁平检索基线 `bm25_rag` 检索并呈现更多黄金支撑
条款。
引用召回率是对此最直接的度量。
它也能抵抗 LLM 对规则自然语言表达的细小变化；这些变化会影响数值匹配指标，却不会改变被引用条款。

**为什么不使用基于冲突的指标。** 一个朴素替代方案是衡量冲突集合表现：给定预测 F 和黄金 F*，对机构
标识符计算集合级 precision、recall 和 F1。
在试点评分中，我们发现该指标在本基准上存在结构性偏差。
我们的黄金抽取会产生机构级冲突标签，并且由于中国高校的科研经费管理政策在几乎每一对机构之间至少有
一个子规则不同，28 个查询的黄金冲突集合包含 12–18 个其他机构。
因此，一个拒绝作出任何具体声明、只是列出所有其他机构的系统，会得到人为偏高的冲突 F1。
我们在 `no_retrieval` 基线中确实观察到这一伪影：它没有来源访问，却达到 0.49 的冲突 F1，超过
`bm25_rag` 的 0.20。
基于引用的指标不允许这种平凡解：要获得引用匹配分数，系统必须输出特定文档中的特定条款编号，而这无法
仅凭机构列表猜出。

### 4.2 基准构建

由于在多文档政策语料上手工标注引用黄金标准成本高昂，我们使用自动抽取流水线生成基准，并明确说明其含义。
对每个（焦点机构，维度）对，我们运行一次专用抽取过程；它只读取该机构在该维度证据窗口中的条款，并要求
LLM 输出：
（i）简短的数值摘要；
（ii）结构化规则组成列表；
（iii）具体支撑引用。
抽取器被约束为只能引用输入证据窗口中出现的条款，这从构造上消除了幻觉条款编号。

过滤前的完整基准包含 6 个维度上的 29 个查询。
其中一个查询的黄金数值是抽取解析错误，我们将其删除，留下本文使用的 28 查询基准。
我们承认，LLM 抽取的黄金标准弱于人工标注黄金标准，并对此作出明确说明。
我们以三种方式缓解风险。
第一，主要指标——(doc_id, article_number) 层级上的引用召回率——是可机械审计的：任何预测引用都可
在来源语料中查找，并判断正确或错误，无需重新审查黄金标准。
第二，抽取器处理的输入严格小于被评估系统的输入：一次只读取一个机构自己的证据窗口。
第 4.2.1 节报告的审计显示，抽取器偏向召回而非精度——它倾向于过度包含主题相邻条款，尤其是在设备和
金额阈值维度上——这会均匀压低每个系统的测得召回率，而不会偏置系统间相对比较。
第三，同一审计显示，当计入部分匹配时，黄金标准精度为 93.0%；按严格标准则为 73.3%。

#### 4.2.1 LLM 抽取黄金标准的人工审计

为回应关于 LLM 抽取黄金引用可靠性的担忧，我们对 6 个代表性查询进行了人工审计，每个政策维度一个；
这些查询被选择为覆盖基准中观察到的系统表现全范围。
所选查询包括：第 4.7 节的一个记录失效案例（`t3a_indirect_cost_fddx_01`）、详细呈现的最佳表现案例
研究（`t3a_labor_cost_hnkjxy`）、其余四个维度中各自表现最好的查询，以及 Wiki RAG 在所有五个查询上
召回率均为零的最差维度（`t3a_horizontal_vs_vertical`）。
审计由作者对照来源政策文件完成，并使用随补充材料发布的结构化审计日志；对每条被审计引用，作者验证
所引条款是否确实支撑归因给它的声明，并分配三种判定之一：正确（条款直接支撑该声明）、部分（条款
主题相关，但只支撑声明的一部分或仅狭义支撑）、错误（条款不支撑该声明）。

表 2 报告了 6 个被审计查询中全部 172 条黄金引用的审计结果。
严格精度（仅计正确）总体为 73.3%（126/172）；当部分匹配也计入时，精度升至 93.0%（160/172）。
只有 12 条引用（7.0%）被判定为完全错误。
按查询看，严格精度范围从 27.6%（设备）到 97.2%（横向与纵向）。
设备维度和金额阈值维度表现出较高的部分匹配率（分别为 18/29 和 13/33），反映了黄金抽取器的召回偏向：
它倾向于包含主题相邻的条款（资产管理、采购流程、固定资产归属），即便这些条款并未严格陈述设备费用
或金额阈值规则。
这种偏差在系统之间是均匀的——每个系统的测得召回率都会被同一因素压低——因此不会偏置本文主要结论
所依赖的系统间相对比较。

12 个完全错误案例集中在五份来源文档上，其中 12 个中的 5 个来自 `fddx_01`（复旦大学，理工医专项文档）。

我们在第 5.2 节和第 6 节讨论未来语义检查层的含义。
由于审计由单一作者完成，我们不报告标注者间一致性；相反，我们发布完整审计工作簿，以便独立审阅者对照

### 4.3 实现细节

所有实验均在一台工作站上运行，配置为 NVIDIA RTX 3090 GPU、Windows 11，以及新建 Anaconda 环境中的
Python 3.10。
所有系统和所有实验阶段使用的语言模型都是智谱 AI 家族的 glm-4-plus，并通过 `zai-sdk` Python SDK 访问。
我们使用 2026 年第一季度可通过智谱 API 获得的模型版本；每批实验的具体 API 快照日期记录在发布的
结果档案中。
评估时温度设为 0（以保证可复现），Wiki 编译时温度设为 0.2（允许在过度约束提示下进行小范围局部恢复，
同时保持表格结构稳定）。
每次 LLM 调用都在本地 SQLite 文件中以 (model, messages, temperature) 层级缓存，因此一旦缓存变热，
重跑任何实验几乎不再产生额外成本；缓存命中时的输出稳定性由构造保证完全一致。
完整代码流水线、基准黄金标准、编译 Wiki、原始评估结果、提示模板、证据窗口关键词列表和分析脚本均作为
单一档案发布（见数据可用性声明）。

### 4.4 主要结果

表 4 报告了三个系统在 28 查询基准上的引用指标，图 3 使用标准差误差条可视化同一组数字。
三个指标上的排序一致：`wiki_rag` 第一，`bm25_rag` 第二，`no_retrieval` 最后。

因此，Wiki 系统相对于扁平 BM25 检索基线的绝对提升为 +0.147，相对提升为 +155%；相对于仅参数化的
下限提升为 +0.219，约为一个数量级。
引用精度也呈同样排序（0.709 > 0.422 > 0.091）：Wiki 系统的优势并非通过不加区分地倾倒更多引用实现，
而是通过选择更多正确引用实现。

在 `wiki_rag` 对 `no_retrieval` 时为大。

三个系统的绝对引用召回率都不高。
一个典型 T3a 查询的黄金引用集合包含跨多个机构的 5–40 条条款；在单一提示中召回其中一半是困难的。

### 4.5 纯检索分析

主要结果数字合并了两个不同组成部分：检索基底的质量（哪些条款被呈现给 LLM），以及 LLM 使用该基底的
质量（在输出中引用了哪些被呈现条款）。
为隔离检索组成部分，我们对每个查询计算 LLM 之前检索集合与黄金引用集合的交集，并在 {1, 3, 5, 10, 20}
上扫描 BM25 的 top-k 参数。
Wiki 系统的 LLM 前检索集合是对应政策维度 Wiki 页面上的完整引用集合；它不依赖 k。

图 4 绘制了 BM25 检索召回率随 k 单调上升的曲线，从 k = 1 时的 0.002 上升到 k = 20 时的 0.244。
Wiki 检索召回率为 0.473，以水平参考线绘制；无检索下限为 0.023，以较低参考线绘制。
在我们测试的范围内，Wiki 检索曲线甚至超过最强 BM25 点：0.473 对 0.244，比值为 1.94×。
结合第 4.3.1 节的上下文预算结果，图景是明确的：Wiki 系统给 LLM 的基底既更小（约为 BM25 token 的
29%），又覆盖更多黄金引用（约为 BM25 检索召回率的 1.94×）。
因此，Wiki 优势是结构性的，源于页面中预编译的跨机构组织，而非更大的上下文体量或更宽的检索。

### 4.6 失效模式分析

为理解每个系统在失败时如何失败，我们将 84 = 28 × 3 个预测中的每一个划分到六个互斥桶之一，并按
自上而下顺序分配：`api-blocked`（内容过滤错误）、`no-citation`（无 `[doc:art.N]` token）、
`hallucinated_cite`（无效机构或条款编号）、`wrong_doc`（所有引用条款结构有效但与黄金无交集）、
`success`（recall ≥ 0.5）和 `partial`（其他情形）。
表 6 报告每个桶的计数，图 5 将同一数据展示为堆叠条形图。

第一，幻觉率按 `no_retrieval` → `bm25_rag` → `wiki_rag` 的顺序单调下降，分别为 75%、54% 和 39%。
加入扁平检索基底使幻觉引用从 21/28 降至 15/28；用结构化 Wiki 替代扁平检索又进一步降至 11/28。

而结构化引用表使该任务显著比非结构化条款拼接更容易。

第二，`wiki_rag` 是唯一产生任何成功输出的系统：它在 3/28 个查询上达到 citation recall ≥ 0.5，
而另外两个系统均为 0。

这一 `wrong_doc` 比例明显高于另外两个系统，后者分别为 0/28 和 2/28。

### 4.7 案例研究：一个查询上的三个系统

为使汇总数字具体化，我们跟踪一个查询在三个系统中的表现。
该查询询问湖南科技学院（`hnkjxy`）对劳务费（劳务费）作何规定，以及哪些其他机构不同。
经对照湖南科技学院政策文件第 14 条验证，真实答案是：在合同未另有约定时，劳务费最高可达实际到位
项目经费的 60%；若干同行机构完全不设此类上限。

`no_retrieval` 基线只接收机构列表和查询，生成了一段关于中国和省级劳务费规定的泛泛叙述，没有具体数字上限。
它引用了五条条款，其中只有一条与黄金集合相交。
五条引用中有三条把规则错误归因给了错误机构。
引用召回率：0.053。

`bm25_rag` 基线接收 BM25 检索得到的前十条条款，找到了 60% 规则，并正确陈述。
然而，对于冲突机构，它只命名了两个：北京大学（`bjdx`）和华东理工大学（`ecust`）。
二者都是正确的，但该查询要求列出所有不同的其他机构，而实际还有许多。
引用召回率：0.158；引用精度：1.000。
该系统准确但不完整。

`wiki_rag` 系统接收表 7 所示的预编译 `labor_cost` 比较页面，也识别出同一 60% 规则。
关键是，它随后以结构化列表列举了另外九个机构的规则，并将九者全部标记为冲突。
该列表包括河南工学院的按学位月度上限（`hngxy`：本科生 ≤ 1000 元/月，硕士 ≤ 2000 元/月，博士
≤ 3000 元/月）、浙江大学和郑州大学的无比例上限制度，等等——每条均来自独立机构自己的政策文件。
系统还正确识别出南开大学政策与湖南科技学院的 60% 上限一致，并将其排除在冲突列表之外。
引用召回率：0.579；引用精度：1.000。

这个例子使汇总数字变得可理解。
三个系统的引用召回率之比为 0.053 : 0.158 : 0.579 = 1 : 3 : 11，与基准整体的 0.023 : 0.095 : 0.242
之比非常接近。
`bm25_rag` 与 `wiki_rag` 的差距并不在于一个找到了焦点规则而另一个没有：二者都找到了。
差距在于，BM25 检索以高相似度呈现焦点机构自身关于所查询维度的条款，而 Wiki 页面呈现同一维度上的
预连接跨机构比较。
Wiki 页面的组织方式将识别冲突机构的任务，从多文档推理问题变成了表格查找。

## 5. 讨论

### 5.1 为什么结构化 Wiki 有助于这一任务族

实验表明，BM25 检索与预编译 Wiki 之间的差距具有一种具体机制；纯检索分析（第 4.5 节）、上下文预算分析
（第 4.3.1 节）和案例研究（第 4.7 节）共同支持这一机制。

BM25 根据查询与每条条款之间的词汇相似度对条款排序。
当查询与相关文章共享有区分度的词汇时，这种方法效果很好，例如当查询命名焦点机构并询问焦点维度时。
事实上，BM25 几乎在每种情形下都检索到焦点机构自身关于查询维度的条款。
困难在于，T3a 查询还有一个被第一项要求掩盖的第二项要求：不仅要陈述焦点规则，还要识别规则不同的
其他机构。
这些其他机构的条款文本中不包含焦点机构名称，并且往往也不共享查询中的具体词汇。
跨机构综合正是 Liu 等人 [4] 发现语言模型在信息分布于长输入上下文中时表现不佳的那类多文档推理。

Wiki 编译器预先执行了这种综合。
编译期间，LLM 读取给定维度上每个机构的条款，并在比较表中为每个机构写入一行，交叉引用已经解析。
查询时，LLM 不再面对长上下文多文档任务；它面对的是单一比较表中的聚焦查找，相关行彼此相邻。
这种预编译在精神上与检索增强预训练 [11] 和迭代检索—生成框架 [27] 有关，但它离线地在文档集粒度上
运行，而不是在线地在 token 或轮次粒度上运行。
纯检索分析从结构上显示了这一点：在我们测量的每个 k 上，Wiki 页面包含的黄金引用都多于任何 top-k
BM25 结果。

失效模式分析（第 4.6 节）从另一个角度强化了这一解释。
加入检索后，幻觉引用从 75% 降至 54%；将检索替换为 Wiki 后进一步降至 39%。
每次下降幅度大致相当（≈21% 和 ≈15%），这与如下解释一致：每一层都以不同程度处理同一类问题，即将
LLM 输出扎根到具体来源文本。
残余 39% 幻觉率显示，即便 Wiki 也不能完全消除该问题。

#### 5.1.1 任务—表示对齐作为适用范围条件

T3a 任务与表格表示天然高度对齐：最直接的人类答案是一张按机构逐行排列的比较表。
我们的 Wiki 页面正是这一表示，并且已经离线预先计算。
因此，我们将第 4.4 节报告的实证差距表述为特定任务族与离线跨文档编译安排之间结构契合的证据，而不是
对检索式方法的一般性优越主张。
自然答案不是表格的任务——例如单文档问答、叙事摘要，或需要超出 Wiki 已编译内容之外推理的查询——
不应被期望从同一安排中获得相同收益。
在我们自己的基准中，`wiki_rag` 在横向与纵向维度上的失败（五个查询全部得分为零）本身就是证据：
即便在单一任务族内部，编译时选择物化哪些维度，也会为查询时系统能够回答什么设定硬性边界。
第 5.2 节讨论这一具体失败。

### 5.2 有效性威胁与讨论

下面列举上述证据的主要限制。

**LLM 抽取黄金标准，仅在子样本上审计。** 基准黄金引用由第 4.2 节描述的 LLM 抽取器生成，而非由人类
专家生成。
我们对 6 个代表性查询中的全部 172 条黄金引用进行了人工审计（第 4.2.1 节，表 2），并发布完整审计
工作簿，使独立审阅者可对照公开来源政策文件复现我们的判断。
该审计由单一作者完成，因此不产生标注者间一致性。
主要数字为：严格精度 73.3%，将主题相邻部分匹配也计入时精度为 93.0%。
审计确认抽取器偏向召回而非精度：它倾向于过度包含主题相邻条款（资产管理、采购流程、合规条款），
即便这些条款并未严格陈述相关规则；但真正错误的引用较少（172 条中 12 条，即 7.0%）。
由于这种偏差在系统之间是均匀的——每个系统的测得召回率都会被同一因素压低——它不会偏置本文主要结论
所依赖的系统间相对比较。
除被审计子样本外，每个预测引用都可以对照来源语料进行机械审计，因此任何对特定比较持怀疑态度的读者
都可以直接重新评估。

**与现代检索系统的关系。** 我们的实验将 `wiki_rag` 与经典 BM25 基线比较。
我们没有实现稠密检索基线（如 DPR [7]、ColBERT [8] 或 BGE-M3 [9]）、稀疏—稠密混合检索或查询扩展方法。
这一范围选择是有意的，原因有二。
第一，本文主要主张是关于以结构化预编译替代扁平检索的效果；加入更多检索变体会成倍扩大实验矩阵，
却不会改变我们试图隔离的问题。
第二，稠密检索器引入 BM25 没有的训练数据依赖，从而使匹配基线设计复杂化。
我们预期稠密或混合检索会缩小 `bm25_rag` 与 `wiki_rag` 之间的部分差距，但同一结构论证仍然成立：
即便经过完美调优的稠密检索器仍返回原始条款，需要 LLM 在查询时交叉引用；而 Wiki 页面返回的是预连接
比较。
一个自然的后续问题是，查询时表格合成（要求 LLM 按需从检索条款中构建比较表）是否会缩小差距；
本文没有测试这一点，但注意到它会在每次查询时承担完整编译成本，而 Wiki 将该成本一次性摊销。
直接的稠密检索对 Wiki 比较，以及按需合成基线，均在第 6 节列为未来工作。

**基于冲突的指标存在结构性偏差。** 如第 3.4 节所述，本基准的黄金冲突集合在每个查询中包含 20 个
可能机构中的 12–18 个。
一个不作任何具体声明、只列出所有其他机构的系统会获得膨胀的冲突 F1。
我们为完整性报告冲突指标，但将其排除在所有显著性检验之外。

**单一 LLM 骨干。** 三个系统均使用 glm-4-plus。
该架构并不内在绑定于这一模型，但我们没有测试跨骨干鲁棒性。
使用不同底层模型进行复现将强化本文主张。

**一个语料、一种语言、一个领域。** 证据限于中国高等教育科研经费管理政策，并限于自然答案结构为表格的
任务族（跨机构规则比较）。
推广到其他表格型跨机构政策领域（如企业合规框架、跨州监管协调）是合理的。
推广到非表格任务则不受本文证据支持，因为在这些任务中，编译时结构可能不匹配查询时信息需求。

**基准规模较小。** N = 28 个查询规模 modest。
我们报告的效应量较大，主要指标上的显著性很强（p < 10⁻³），但若基准规模扩大三倍，将进一步降低结果风险。

**Wiki 编译会产生引用检查器无法捕捉的幻觉。** 引用检查器验证每个 `[doc:art.N]` token 是否指向有效
文档和有效条款编号，但不验证周围文本与所引条款之间的语义对齐。
我们编译 Wiki 中两个具体失败说明了这一点。
第一，在 `indirect_cost` 页面中，一行把机构 `sysfdx`（沈阳师范大学）标为华中科技大学。
引用本身结构有效（`[sysfdx:art.19]` 指向沈阳师范大学文档中的真实条款），但展示名称是 LLM 幻觉。
第二，在关于复旦大学理工医专项文档（`fddx_01`）的 `indirect_cost` 查询中，`wiki_rag` 系统在答案开头
陈述了属于安徽工业大学（`ahgydx`）的规则，并正确引用 `[ahgydx:art.12]`。
模型将正确引用附着到了错误的焦点机构上——这是焦点混淆失败；即便引用元组本身技术上有效，它也会落入
我们的 `wrong_doc` 桶。
这两类失败都说明，需要超越当前流水线的语义检查层。
一种自然设计是在声明文本与所引条款正文之间加入基于 NLI 或 LLM-as-judge 的一致性检查；ALCE [12]
框架和 Self-RAG [22] 的反思 token 机制是我们在第 6 节列出的具体候选方向。

**API 内容过滤交互。** 中文科研经费政策语料中的一些段落包含治理相关词汇（审计、纪律、违规），在较高
检索深度下会触发底层 API 的内容安全过滤器。
在默认 BM25 k = 10 下，我们观察到零次此类阻断，因此主要结果不受影响；在 k = 20 的探索性运行中偶尔
失败。
若有影响，也是在较大 k 下不利于 BM25，而不是有利于 `wiki_rag`。

**高风险部署含义。** 最佳系统达到的绝对召回率 0.242 意味着，在典型 T3a 查询上，系统呈现约四分之一
相关支撑条款。
对于研究助理或文献筛选应用，只要人工审阅引用证据是工作流的一部分，这就是有用的一阶工具：
每条被呈现引用要么支撑声明，要么很快无法通过人工验证。
对于自动化合规、审计或科研经费管理决策支持，剩余四分之三遗漏引用是严重风险：一个自信列出部分冲突
机构规则的系统，可能给用户造成完整性的错觉。
因此，我们将当前系统定位为适合人机协同政策探索，而不是可部署的端到端自动化。
缩小召回差距是后一种用例的前提。

## 6. 结论

本文在 28 个跨机构政策冲突查询上，对扁平 BM25 检索增强基线与预编译 Wiki 检索基底进行了实证比较；
这些查询来自中国高校科研经费管理文档。
在主要指标引用召回率上，Wiki 系统达到 0.242 ± 0.21，而 BM25 为 0.095 ± 0.12，仅参数化基线为
0.023 ± 0.03；在配对 Wilcoxon 检验下，成对差异均达到 p < 10⁻³ 的统计显著性。
我们将这一结果表述为关于跨文档比较任务与“先编译、后查询”安排之间结构契合的设计启示，并以概念验证
实现作为支撑，而不是相对于现代检索系统的一般性优越主张。

三项次级分析细化了这一解释。
纯检索测量显示，Wiki 页面包含的黄金引用多于 k = 20 时的 BM25，因此优势是结构性的，而不是检索广度
问题；这一发现与关于 LLM 增强系统中检索设计选择的更广泛观察一致 [26]。
上下文预算测量（第 4.3.1 节）显示，Wiki 系统是在仅使用 BM25 输入 token 约 29% 的情况下达到该效果的，
排除了 Wiki 只是接收更多信息这一替代解释。
失效模式分类显示，幻觉引用率在三个系统间从 75% 降至 54% 再降至 39%，这与每次检索升级都在处理同一
根本问题——将 LLM 输出扎根到具体来源文本——相一致。
对 6 个代表性查询（每个政策维度一个；第 4.2.1 节）中全部 172 条黄金引用的人工审计显示，当计入部分
匹配时，LLM 抽取黄金标准的精度为 93.0%（严格为 73.3%）；12 个完全错误案例集中在五份文档上，并可
追溯到反复出现的语义归因失败模式，这也引出了第 6 节中的未来工作方向。

结果的范围是有限的。
它限于一种语言、一个语料、一个生成器，以及一个自然答案为表格的任务族。
任何系统的绝对召回率都不高，而且编译 Wiki 本身包含引用级验证器无法捕捉的失败。
本文贡献是在一个特定任务上的受控实验，而不是关于如何为 LLM 消费安排知识的普遍处方。

从实验已经显示和尚未显示的内容出发，可以提出四个未来工作方向。
第一，能够验证被引条款与周围声明之间对齐关系的语义引用检查——例如基于 NLI 的过滤器，或借鉴 ALCE
框架 [12]、Self-RAG 反思 token [22] 的 LLM-as-judge 一致性检查——将处理本文报告的焦点混淆和展示
名称失败。
第二，稠密检索基线（例如使用 bge-m3 [9] 或 LLM 微调多阶段检索器 [25]）以及查询时表格合成基线，将
有助于从结构化预编译本身的贡献中，隔离稀疏与稠密检索、编译时与查询时结构化的贡献。
第三，在第二个语料上复现——可能使用不同语言或不同监管领域——将检验本文观察到的结构优势是否可在
跨机构比较任务族内部泛化。
第四，更大且至少部分由人工标注的基准将收紧绝对 precision 和 recall 数字，并减少围绕 LLM 抽取黄金
标准的剩余不确定性。
