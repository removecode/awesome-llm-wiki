# WikiKV 检索方式说明

分两层理解：**WikiKV** 讲的是「怎么存、怎么导航取页」；完整「智能体怎么推理式检索」在姊妹篇
[Retrieval as Reasoning](https://arxiv.org/abs/2605.25480)。

## 1. 和普通 RAG 的对比

| | 普通 Dense RAG | 这篇体系 |
|--|--|--|
| 知识长什么样 | 切成扁平 chunk + 向量库 | 先编译成有层次、可链接的 wiki |
| 检索一次做完？ | 通常一次 top-k 就交给生成 | **多步**：搜 → 读 → 不够再扩/再跟链接 |
| 主要靠什么定位 | 向量相似度 | **路径/标题等文本信号**（向量不是主路径） |

WikiKV 原文要点：路由器作用在**文本路径键**上，不是稠密向量；若使用向量检索，也只限于叶内容相关算子，并与路径路由正交。

## 2. WikiKV 中的检索算子：`NAV(q, B)`

- **输入**：自然语言问题 `q` + 墙钟时间预算 `B`（毫秒）
- **输出**：一串**从粗到细**的结果；时间不够也能返回已拿到的前缀（anytime / 渐进语义）

### 2.1 流程（对应论文算法 1）

```text
用户问题 q
    │
    ▼
① CLASSIFY(q)          ← 很快：正则 + 小分类器（≤5ms）
    │
    ├── 是「枚举类」     （哪些维度？列表一下…）
    │       └─→ 只做 LS("/")  列根目录 → 结束
    │
    └── 其他（要找具体内容）
            │
            ▼
② EXTRACT(q)           ← 从问题里抽出可能的页面名关键词
            │
            ▼
③ SEARCH(...)          ← 在「路径命名空间」上做关键词/前缀搜索
            │              取 Top-k（论文里 k=3）候选路径
            │
            ▼
④ 对每个候选路径：
      GET(路径)          ← 读出该页
      若 NEEDSDEEPER     ← 这页还盖不住问题？
         再 LS(路径)     ← 只往下扩一层（列子节点）
      预算用尽就停
            │
            ▼
⑤ 返回已累积的页面序列 R（供后面生成答案）
```

### 2.2 四个动作分别是什么

1. **CLASSIFY**  
   判断问题是「列清单」还是「找某件事」。  
   枚举类不用搜，直接看根目录就够。

2. **SEARCH（核心）**  
   不是 embedding 近邻，而是：从问题抽词 → 在路径/页面名上做**前缀/关键词扫描** → 直接跳到树里大致正确的位置。  
   这样不必从根开始，每层问一次 LLM「走左边还是右边」。

3. **GET / LS**  
   存储原语：按路径取一页，或列出某节点的孩子。

4. **NEEDSDEEPER**  
   看当前页内容够不够回答；不够再扩一层。可以是小分类器，也可以是一次 LLM 调用。

### 2.3 「渐进答案」长什么样

按层次往外吐：

1. 最粗：整个 wiki 有哪些维度  
2. 再细：某维度下有哪些实体  
3. 更细：具体实体页 / 文章内容  

预算到了就停——前面那些仍然算有效答案，只是更粗。

### 2.4 和「逐层 LLM 下树」比

- 笨办法：每层一次 LLM，深度 5 ≈ 5 次串行 LLM  
- 这篇：先 SEARCH 定位，后面通常只要 0～1 次「要不要再深一点」  
- 论文结论：期望 LLM 辅助下降从 \(O(\text{depth})\) 降到大约 \(O(1)\)

## 3. 姊妹篇中的智能体检索

WikiKV 写明：端到端问答流水线承接自先前工作。那边把检索做成 **ReAct 式工具循环**，不是单次 `NAV` 黑盒：

| 工具 | 作用 |
|------|------|
| `wiki_search(query)` | 搜索引；优先页面名、别名、标签、描述，再才退回正文 |
| `wiki_read(paths)` | 读目录或整页；页里的 wikilink 供下一步跳转 |

智能体大致三种走法：

- **直接**：已知实体 → 搜一下再读，或直接读  
- **桥接**：读 A → 跟链接到 B → 再组合证据（多跳）  
- **浏览**：开放题先读目录概览，再挑有希望的页读  

每读完一页会判断「证据够不够」；够了才回答，或预算/耐心用尽就停。

### 两层关系

```text
[19] 应用层：智能体循环调用 search/read/跟链接
        ↓ 落在
WikiKV 存储层：GET / LS / SEARCH(路径) / 缓存 / 一致性
        其中 NAV ≈ 把「搜索加速导航」收成一个带预算的算子
```

实验里的 `LLM-Wiki (WIKIKV)`，就是这套应用检索跑在 WikiKV 存储上。

## 4. 小例子

问题：「周作人和鲁迅什么时候失和？」

1. CLASSIFY → 不是枚举，是定位  
2. EXTRACT → 关键词如「周作人」「鲁迅」「失和」  
3. SEARCH → 命中路径如 `/人际关系/论敌/...` 或相关实体页（Top-3）  
4. GET 读这些页；若摘要不够，LS 再看子页或跟到 `/sources/articles/...`  
5. （若走姊妹篇智能体）还可能读页内链接、再 `wiki_search` 补一跳  
6. 把读到的页交给生成模型写最终答案  

## 5. 一句话总结

**检索方式 = 在编译好的 wiki 树上做「路径/标题搜索 + 读页 + 必要时扩一层/跟链接」，带时间预算；主路径不是向量 top-k。**

- WikiKV：把「搜索加速导航」写成 `NAV(q, B)`  
- 姊妹篇 [2605.25480](https://arxiv.org/abs/2605.25480)：更完整的多轮 `wiki_search` / `wiki_read` 智能体循环
