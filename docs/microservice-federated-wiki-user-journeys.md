# 微服务联邦知识库：用户故事与用户旅程

## 0. 这份文档要解决什么问题

`microservice-federated-wiki-design.md`（下称"主文档"）从数据模型出发，先定义 K0–K4、覆盖等级、
检索深度、统一信封，再倒推场景（7.3 节）。这个顺序对模型的一致性有利，但会让人看不出"哪一块是被
真实场景逼出来的，哪一块是先把模型建完、场景只是事后补的说明"，容易显得大而全。

本文档反过来走：先列用户故事，再串成端到端用户旅程，最后回过头核对主文档每一节是否被至少一条旅程
实际用到。不重新定义模型——K0–K4、C0–C3、D0–D3、`execution_authority`、`procedure`/`project` 等
术语全部沿用主文档；本文档只做"落地视角的取舍"，产出一份"按场景做，不按分层做"的核对清单。

本文档同时明确区分两个容易混在一起的维度：**角色**（是谁，身份和决策权不同）和**场景**（同一个角色
在不同情境下面临的具体任务）。第 1 节说明判断标准，避免角色列表随着场景增多而膨胀。

## 1. 角色与场景

### 1.1 判断标准

角色和场景经常被混着列成一张扁平表，导致"角色"数量随场景增多而不断膨胀。区分标准很简单：**换一个
人来做，结果是否本质不同**。

- 如果换成另一个人做同一件事，只是"这次问的问题不一样/这次要交接的项目不一样"，本质上还是同一批
  人在不同时刻的不同任务——这是**场景**，不是角色。
- 如果确实是不同的人在做（不同团队、不同目标、不同决策权、汇报关系不同），才是**角色**。

按这个标准重新核对：后端/业务工程师问答和让 Claude Code 编码，决策权始终在同一个工程师身上，
应合并为一个角色；"项目交接负责人"只是这个工程师在管理一个项目全生命周期（立项、推进、交接、
收尾）过程中面临的一个场景，再抽象一层就是"项目管理场景"，交接只是其中一个环节，不是全部；
"新成员/跨域协作者"也只是这个工程师接触陌生服务/域时面临的场景。值班工程师/SRE 保留为独立角色，
因为值班/SRE 团队与写业务代码的工程师不是同一批人、决策权也不同。

### 1.2 角色（数量应保持很少）

| 角色 | 是谁 | 目标 | 决策权归属 |
|---|---|---|---|
| 工程师（人 + Claude Code） | 负责服务开发/维护的工程师；问答还是让代理干活，决策权都在此人 | 完成开发、定位问题、确认设计、交接工作 | 研发团队 |
| 值班工程师 / SRE | 独立的值班/SRE 团队成员，不一定写业务代码 | 快速定位并处置线上故障，判断是否升级 | SRE/值班团队 |
| 知识库维护者 / 架构师 | 治理知识库本身的岗位：编译质量巡检、业务域划分确认 | 保证知识库正确性，推动业务域命名共识 | 平台/架构团队 |

三个角色对应主文档 7.3 节的三大类场景分类（编码/问答由"工程师"承担，故障处理由"值班工程师/SRE"
承担，Error Book 与域划分确认由"知识库维护者/架构师"承担）。

### 1.3 场景（挂在角色之下，可以持续增长）

"工程师"角色下目前有四类场景，都是同一批人在不同情境下的任务，不是不同的人：

- **日常问答场景**：想确认一个事实再动手（US-5、US-6）。
- **编码场景**：让 Claude Code 完成一个编码任务（US-3、US-4）。
- **新人/跨域上手场景**：第一次接触陌生服务或域、要走组织级规程、或想了解某业务线全景
  （US-11、US-12、US-13、US-14）。
- **项目管理场景**：管理一个项目从立项到收尾的全过程——记录背景/目标/进度、随时查询状态、
  会话或排班结束时交接、项目结束时关闭（US-7、US-8、US-15）。之前把这个场景窄化成"项目交接
  场景"，只看到"交接"这一个触发时刻，抽象成"项目管理"后才看到查状态、记进度这些不涉及交接的
  子场景，见 US-15 和旅程 H。

值班工程师/SRE 和知识库维护者/架构师目前各自只对应一类场景（故障处理；编译质量巡检 + 域划分
确认），暂不需要再往下拆，等出现更多真实故事再决定要不要拆分。

## 2. 用户故事（User Stories）

格式统一为：作为 <角色>，当 <场景触发> 时，我想要 <诉求>，以便 <收益>。每条附验收标准，用于后续
判断"主文档里支撑它的那部分是不是刚好够用，还是明显过度设计"。

### 2.1 工程师 —— 日常问答场景

- **US-5**：作为工程师，当我只是想确认"这个接口幂等吗"时，我想要一次调用就拿到答案和引用，
  以便不用被迫加载整个服务的架构说明。
  验收标准：默认止步 D1–D2，不强制沿关系跨包展开。
- **US-6**：作为工程师，当我看到两份文档对同一个设计决策说法不一致时，我想要看到两条 claim
  以及各自证据，而不是被系统偷偷合并成一个"正确答案"。
  验收标准：冲突 claims 必须原样并列返回，各自标来源。

### 2.2 工程师 —— 编码场景（由 Claude Code 执行）

- **US-3**：作为工程师，当接到"订单偶尔重复创建，定位并修复"这类任务时，我想要 Claude Code 从
  故障现象直接跳到可疑代码符号和证据，以便不用先读完整份架构文档才能定位问题。
  验收标准：能落到 `repo@commit:path#symbol` 级别（D3），且证据可追溯到 K0。
- **US-4**：作为工程师，当要修改一个被多个服务消费的共享字段时，我想要先拿到候选影响面，再逐个
  受影响服务补充细节，以便不会漏改下游，也不会一次性拉爆上下文预算。
  验收标准：`analyze_change_scope` 先返回候选列表，不自动展开到 D2–D3；展开必须逐服务、有界。
- **US-16**：作为工程师，当 Claude Code 编码过程中需要调用某个内部 SDK/公共能力组件（比如统一
  RPC 客户端、日志 SDK、鉴权中间件）时，我想要自动带出这个 SDK 的用法说明和常见坑，而不是等我
  手动去问"这个 SDK 怎么用、文档在哪"。
  验收标准：代码里的依赖/import 能关联到对应的 `shared_capability` 实体后，应主动附带其 K2 用法
  说明；这是"主动加载"而不是"等一次新的问答触发"，与 US-3/US-4 的被动查询模式不同，见旅程 I。

### 2.3 工程师 —— 新人/跨域上手场景

- **US-11**：作为工程师，当我第一次接触一个陌生服务时，我想要先看职责和边界，再决定要不要往下看
  接口细节，以便不被一次性塞进大段代码符号。
  验收标准：默认返回 K2 `overview`，只做实体解析，不主动扩图（对应主文档 D0–D1）。
- **US-12**：作为要接入新服务的工程师，我想要看到同域参照服务在鉴权/日志/配置上的既有约定，以便
  新服务能对齐规范而不是自己再发明一套。
  验收标准：能定位到 `shared_capability` 和至少一个同域参照服务的 K2 全量。
- **US-13**：作为工程师，当我要为服务申请上线资质（比如安全审查、权限开通）时，我想要查到一份
  组织级的申请流程（不挂在任何具体业务域下），包含每一步该找谁、需要什么材料，以便不用到处问人
  才能走完流程。
  验收标准：`procedure` 打 `scope: org` 标签、`coverage_level` ≥ C2 时才能作为可执行步骤返回；
  每步 `actor` 必须写明谁负责（审批人/经办人），不能只写"做什么"。
- **US-14**：作为工程师，当我想了解"支付业务线整体是怎么运转的"时，我想要先看这条业务线包含哪些
  服务及其角色（主责/支撑/共享），再看几条关键业务流程的步骤，以便建立全局认知，而不是从零散的
  单服务文档里自己拼。
  验收标准：能通过 domain 关系一次性列出服务清单与角色；已确认 flow 与候选路径必须分开展示，不能
  把未确认路径当成"业务线整体流程"直接呈现。

### 2.4 工程师 —— 项目管理场景

项目管理覆盖立项、推进中持续记录、状态查询、交接、收尾五个环节；交接（US-8）只是其中一个环节，
不能代表整个场景，US-15 就是抽象成"项目管理"之后才浮现出来的、不涉及交接的子场景。

- **US-7**：作为工程师，当我在推进一个项目时，我想要把背景（为什么启动）、目标（要达成什么）、
  进度（做到哪一步、卡在哪）这些过程信息持续记录进项目包，而不是只在最后一次会话结束时才补一份，
  以便项目在任意时间点被交接出去时，接手的人不用再去问"这个项目是干什么的、现在做到哪了"。
  验收标准：项目包必须能承载背景/目标/进度三类信息并可持续更新（不只是收尾时补的 `open_items`/
  `session_anchor`）；每次推进后应更新，不能只依赖"下班/结束会话"这一个触发点。
- **US-8**：作为接手项目的工程师，当我打开一个项目包时，我想要按 `includes` 自动带出相关服务和
  流程的摘要，而不是拿到一堆 ID 自己再去逐个查。
  验收标准：`get_project_context` 一次返回范围清单 + 摘要 + 陈旧警告，默认 D1–D2。
- **US-15**：作为工程师，当我只是想知道"这个项目现在进展到哪了"（不涉及接手，比如自查或者向上
  汇报前先确认一下），我想要直接读到最新的背景/目标/进度，而不需要触发一次完整的交接流程。
  验收标准：查询进度和"接手前预读"必须能复用同一份项目包数据，不需要为"只是问一下"单独走一遍
  只有交接场景才需要的协议（比如强制刷新 `session_anchor`）。

### 2.5 值班工程师 / SRE

- **US-1**：作为值班工程师，当收到 `payment_error_rate_high` 告警时，我想要立刻拿到一份带步骤、
  带执行风险标注的应急预案，以便知道哪几步能自己先跑、哪几步必须叫醒团队负责人。
  验收标准：返回结果必须区分 `ai_assistable`/`human_approval_required`/`human_only`，且低于 C2 的
  候选预案要显式标"未确认"，不能直接当可执行步骤用。
- **US-2**：作为值班工程师，当预案执行到升级条件（15 分钟未恢复）时，我想要自动看到下一步该
  升级给谁，以便不用翻聊天记录找联系人。
  验收标准：`escalates_to` 指向的下一个 `procedure` 能一次性带出。

### 2.6 知识库维护者 / 架构师

- **US-9**：作为架构师，当同一类编译错误反复出现时，我想要知道这个错误上次是怎么被修的、约束
  有没有被注入到下一次编译提示词里，以便判断要不要升级为强制规则。
  验收标准：Error Book 条目要带现象、根因约束、生命周期状态（open/closed）。
- **US-10**：作为架构师，当算法给出一批候选业务域划分时，我想要能清楚区分"算法建议"和"负责人已
  确认"，以便不会把未经确认的社区发现结果当成事实传播出去。
  验收标准：候选标签与已确认业务域在展示上必须可区分，不能静默提升置信度。

## 3. 用户旅程（端到端）

每条旅程给出：触发 → 用户动作 → 命中的知识层/MCP 调用 → 决策点 → 结果/异常路径，用于检验主文档的
分层和 MCP 方法是否刚好够串完一次真实流程，而不是纸面上自洽但实际拼不起来。旅程标题标注"角色 +
场景"，不再单独用场景名当角色。

### 旅程 A：值班工程师/SRE 处理支付故障（对应 US-1、US-2）

| 阶段 | 用户动作 | 命中的知识/调用 | 决策点 |
|---|---|---|---|
| 1 触发 | 收到 `payment_error_rate_high` 告警 | — | 是否已有对应预案 |
| 2 定位预案 | 查"支付失败怎么排查" | `resolve_entity` → `get_procedure(...)` | `coverage_level` 是否 ≥ C2 |
| 3 执行诊断步骤 | 按步骤 1 检查健康检查和部署记录 | 该步 `execution_authority: ai_assistable`，代理可自动执行 | 结果指向步骤 2 还是步骤 3 |
| 4 执行处置步骤 | 回滚最近一次发布 | 该步 `execution_authority: human_only`，代理只能提示 | 团队负责人是否确认执行 |
| 5 未恢复升级 | 15 分钟错误率未回落 | `escalates_to` → `procedure:payment-major-incident` | 是否需要拉更大范围的应急 |
| 6 收尾 | 事故复盘 | 若本次预案首次被真实事故验证，`coverage_level` 由 C2 提升到 C3 | 是否需要更新 `review_cadence_days` |

旅程验证：`execution_authority` 按步骤标注、`escalates_to`、`coverage_level` 的 C2→C3 演进都在
这一条旅程里被实际用到，不是纯理论设计。

### 旅程 B：工程师 —— 编码场景，修复"订单重复创建"（对应 US-3、US-4）

| 阶段 | 用户动作 | 命中的知识/调用 | 决策点 |
|---|---|---|---|
| 1 接任务 | 收到 bug 报告 | — | 先问答定位还是直接改代码 |
| 2 定位现象 | 查"订单重复创建"相关知识 | `search_knowledge` → K2 `operations.md` + failure_mode | 是否命中已知故障模式 |
| 3 落到代码 | 需要具体符号 | `get_service_context` → `get_evidence` 到 `repo@commit:path#symbol` | 是否只是幂等键缺失这类已知模式 |
| 4 评估影响面 | 改动前确认下游 | `analyze_change_scope`，只返回候选影响面 | 是否需要人工确认后才展开 D2–D3 |
| 5 逐个核实 | 对候选服务补 `get_service_context` + `get_evidence` | 有界展开，不无限传播 | 每个候选是否真的受影响 |
| 6 提交修复 | 改代码 + 更新关联 facts | Git commit 触发 K1 更新，标记关联 section 陈旧 | 是否需要人工复核编译结果 |

旅程验证：K0–K2 之间的跳转、`analyze_change_scope` 的"候选而非确定"语义、以及 9 节的陈旧标记
机制，在这条旅程里都是必需环节，没有一步是多余的。

### 旅程 C：工程师 —— 项目管理场景，推进并交接"结算延迟优化"（对应 US-7、US-8）

| 阶段 | 用户动作 | 命中的知识/调用 | 决策点 |
|---|---|---|---|
| 1 立项 | 创建项目包，写清背景和目标 | 初始化 `handoff.yaml`（目标/范围）+ `wiki/overview.md`（背景与目标） | 目标是否可衡量 |
| 2 推进中记录进度 | 每完成一个阶段性结论就更新一次，不等收尾才写 | 更新 `wiki/status.md`（进度/阻塞/风险） | 是否需要同步补 `open_items` |
| 3 接手 | 打开项目包 | `get_project_context(checkout-latency-opt)` | 是否有陈旧警告需要先处理 |
| 4 补齐背景 | 按 `includes` 拉服务/流程摘要 | 各 service/flow 的 D1–D2 摘要 | 是否需要深入到某个服务的 D3 |
| 5 继续排查 | 沿 `open_items` 里的线索往下查 | 定位到具体 `endpoint` 证据 | 是否排除了某个假设 |
| 6 会话结束 | 回写状态 | 更新 `handoff.yaml` 的 `open_items`/`session_anchor` | 是否遗漏未决事项 |

旅程验证：这条旅程暴露一个和 US-7 相关的更新节奏问题——主文档 6.5 节只写了"会话结束时应回写
`handoff.yaml`"，容易被理解成"只在收尾这一个时间点写"；但背景/目标在立项时就要写（步骤 1），
进度应该随每个阶段性结论持续更新（步骤 2），不是攒到最后一次会话结束才补。建议在 6.5 节把更新
时机明确成"立项时写背景与目标、进展中持续更新 `status.md`、会话结束时才回写 `open_items`/
`session_anchor`"三个节奏，而不是笼统一句"会话结束应回写"。另外，如果某次交接完全不需要
`session_anchor`（比如任务已经彻底做完在关闭项目），说明 `project` 包在"收尾"场景下可以只写
`status.md` 为 closed，不强制填 `session_anchor`，这点仍建议在 6.5 节补一句说明。

### 旅程 D：工程师 —— 新人/跨域场景，问答"订单服务是做什么的"（对应 US-11、US-5）

| 阶段 | 用户动作 | 命中的知识/调用 | 决策点 |
|---|---|---|---|
| 1 提问 | "订单服务是做什么的" | `resolve_entity` → K2 `overview` | 是否需要继续深入 |
| 2 追问 | "它依赖哪些服务" | `traverse_relations(predicates=[calls], depth=1)` | 是否要看具体接口 |
| 3 结束 | 得到答案 | 全程止步 D0–D1，未触发关系跨包展开 | — |

旅程验证：这是最短的一条旅程，恰好说明 K3/K4 的很多能力（flow、procedure、project）在这类最高
频的问答场景里完全不会被触发——这正是判断"哪些部分容易被过度设计"的信号，见第 4 节。

### 旅程 E：知识库维护者/架构师巡检 Error Book（对应 US-9、US-10）

| 阶段 | 用户动作 | 命中的知识/调用 | 决策点 |
|---|---|---|---|
| 1 巡检 | 查本周新增的编译错误 | `get_error_book(filters={status: open})` | 是否是同一类错误重复出现 |
| 2 溯源 | 查错误关联的 section/实体 | Error Book 条目引用的 K1/K2 | 根因是抽取器问题还是文档本身问题 |
| 3 决策 | 是否升级为强制约束 | 更新条目并标记是否已注入下一次编译 prompt | 是否需要人工复核修复效果 |
| 4 域划分复核 | 查看算法候选业务域 | K3 domain 候选标签 vs 已确认标签 | 是否找负责人确认 |

### 旅程 F：工程师 —— 新人/跨域场景，查询"新服务安全审查申请流程"（对应 US-13）

| 阶段 | 用户动作 | 命中的知识/调用 | 决策点 |
|---|---|---|---|
| 1 提问 | "新服务上线要做安全审查，怎么申请" | `resolve_entity` → 命中 `procedure` 且 `scope: org` | 是否已有对应组织级规程 |
| 2 拿到步骤 | 查看步骤与经办人 | `get_procedure(security-review-application)` | `coverage_level` 是否 ≥ C2 |
| 3 判断可执行性 | 看每步 `execution_authority` | 多数是 `human_approval_required`（提交材料/等审批） | 是否需要人推进 |
| 4 未找到 | 若查不到对应规程 | 显式返回"无覆盖"而不是编造步骤 | 是否需要人工补充这份规程 |

旅程验证：这条旅程说明"组织级通用规程"（主文档 6.4 节 `scope: org`）不是可选项，而是工程师在
新人/跨域场景下会高频问到的一类问题；第 4 节里之前把它标为"暂缓投入"需要修正，见下节更新。

### 旅程 G：工程师 —— 新人/跨域场景，了解"支付业务线"整体流程（对应 US-14）

| 阶段 | 用户动作 | 命中的知识/调用 | 决策点 |
|---|---|---|---|
| 1 提问 | "支付业务线整体流程是什么" | `resolve_entity` → 命中 `domain:payment` | 是否存在已确认的业务域 |
| 2 列服务清单 | 看业务线包含哪些服务 | `traverse_relations(domain:payment, [includes])` 带出角色 | 是否按角色过滤 |
| 3 看关键流程 | 看几条关键业务流程 | 对域下已确认的每个 `flow_id` 调 `get_flow` | 已确认流程还是候选路径 |
| 4 深入某一步 | 追问某个服务的细节 | `get_service_context(service_id, aspects=overview)` | 是否需要继续深入到 D2 |

旅程验证：这条旅程能跑通，但暴露主文档一处规格不对称——不是"K3 这一层不做 LLM 编译"（主文档
1 节把 Claims 定义为"LLM 基于事实编译出的职责、架构、约束、**流程**和故障模式"，K3 章节标题也叫
"跨服务业务**知识**层"，按定义应该和 K2 一样有编译产物），而是 `procedure`（6.4 节）和 `project`
（6.5 节）都写明了目录结构、编译产物文件名（`wiki/runbook.md`、`wiki/overview.md` 等），domain
（6.2 节）和 flow（6.3 节）都没有写这一层规格；domain 甚至连专属 MCP 方法都没有（8 节只有
`get_flow`，没有 `get_domain_context`），只能靠 `resolve_entity` + `traverse_relations` 现场拼。
建议主文档在 6.2/6.3 节补上目录结构和编译产物文件名，domain 再补一个 `get_domain_context` 方法，
而不是继续靠组合调用现场拼装。这条旅程也顺带印证了一个不需要新设计的点：域级"简略概览"与
服务级"详细接口/业务规则"本来就该是不同深度的产物，这正是主文档 D0–D3 检索深度和 K2 分页
（`overview.md` 简略、`interfaces.md`/`architecture.md` 详细）已经覆盖的模型，不用为"问答用简略
文档、编码用详细文档"这条经验再单独立一条用户故事。

### 旅程 H：工程师 —— 项目管理场景，只查进度不交接（对应 US-15）

| 阶段 | 用户动作 | 命中的知识/调用 | 决策点 |
|---|---|---|---|
| 1 提问 | "结算延迟优化那个项目现在做到哪了" | `get_project_context(checkout-latency-opt)` | 是否只需要 D1 摘要 |
| 2 读取 | 拿到背景/目标/`wiki/status.md` 里的最新进度 | 复用旅程 C 里持续更新的同一份数据 | 是否需要看具体某条 `open_item` |
| 3 结束 | 得到答案，不写回任何字段 | 只读，不触发 `session_anchor` 更新 | — |

旅程验证：这条旅程之所以在之前的版本里没被发现，是因为场景被窄化成"项目交接"——只想着"谁接手
时要看什么"，没想到"不接手、只是想看一眼"也是同一份数据的合法读法。抽象成"项目管理场景"之后，
这类"只读不写"的查询才被显式列出来，说明 `get_project_context` 需要支持"只读进度"和"接手预读"
两种意图共用同一次调用，不应该只为交接场景设计。

### 旅程 I：工程师 —— 编码场景，调用内部 RPC SDK 时自动带出用法（对应 US-16）

| 阶段 | 用户动作 | 命中的知识/调用 | 决策点 |
|---|---|---|---|
| 1 接任务 | "给下单接口加一次对库存服务的调用" | — | 需不需要引入新的 SDK/客户端 |
| 2 识别依赖 | Claude Code 发现要引入内部统一 RPC 客户端 | 依赖名匹配到 `shared_capability:rpc-client` | 是否已有对应记录 |
| 3 主动加载用法 | 不等工程师问，直接带出这个 SDK 的用法 | `get_service_context(internal-rpc-client, aspects=interfaces)` | 用法说明是否够写出正确调用代码 |
| 4 未命中 | 若代码里的依赖没关联到任何 `shared_capability` | 显式提示"未找到该依赖的使用规范"，不能瞎猜用法 | 是否需要人工补充这份 SDK 说明 |
| 5 写代码 | 按拿到的用法写调用代码 | 沿用旅程 B 的证据核实流程 | 是否需要再核实一次接口签名 |

旅程验证：这条旅程和旅程 B、F 都不一样，之前的旅程全是"工程师先问，MCP 再答"的被动查询模式；
这里的触发点是"代码里出现了一个依赖"，知识应该跟着这个信号主动加载，而不是等一次新的自然语言
提问。主文档目前的 MCP 方法（8 节）和上下文组装流程（7.4 节）都是按"收到问题 → 返回结果"设计
的，没有覆盖"编码过程中检测到依赖变化就主动推送相关知识"这种模式，这是本轮新发现的一处交互
模式空白，建议下一轮评估要不要在 `get_service_context` 之外补一个"按依赖反查 `shared_capability`"
的轻量路径，而不是要求工程师每次都手动问一遍。

## 4. 从旅程反推设计取舍

对照九条旅程和十六条用户故事，逐一核对主文档各节是否被实际用到：

| 主文档能力 | 被哪条旅程验证 | 结论 |
|---|---|---|
| K0–K2 分层、section 检索 | A、B、C、D、E、F、G、H、I 全部用到 | 核心能力，必须保留 |
| `execution_authority` 按步骤标注 | A、F | 高频且关键，保留 |
| `analyze_change_scope` 候选影响面 | B | 编码场景必需，保留 |
| `coverage_level` C2→C3 演进 | A | 保留，但只在 procedure 上验证过，flow 的 C3 演进暂无旅程覆盖 |
| `project`/`handoff.yaml` | C、H | 保留，但更新节奏需拆成立项/进展中/收尾三段；查询需与交接解耦（见 H） |
| K3 flow（跨服务业务流程） | G（此前只在 B 的背景信息里间接出现） | 已被独立旅程验证，纳入核心；目录结构/编译产物规格需补齐 |
| K3 domain 服务角色关系（primary/supporting/shared） | G | 已被独立旅程验证，纳入核心，不再是"轻量可选"项 |
| Error Book | E | 保留，但目前只有架构师角色使用，暂不需要为其他角色单独优化交互 |
| `procedure` 的组织级通用规程（`scope: org`） | F | 已被真实用户故事验证，从"暂缓投入"改为核心，需要保证 `actor` 字段可用 |
| domain 的目录结构、编译产物文件名、专属 MCP 方法 | G 暴露的空白，主文档未定义 | 新发现的规格缺口，建议直接补充，不是能力缺失 |
| `shared_capability`（内部 SDK/公共能力） | I | 已被独立旅程验证，但"主动加载"这个触发模式主文档没设计 |

由此得到三类结论，呼应"从大而全收缩到按场景"的诉求：

- **已被多条旅程验证、应该保留的核心**：K0–K2 三层、section 级检索、`execution_authority`、
  `analyze_change_scope` 的候选语义、`procedure` 的执行风险分级和组织级规程、domain 的服务角色
  关系。这些是不能再削减的最小集合。
- **仍缺旅程支撑、可以继续暂缓的部分**：flow 的 C3（演练验证）演进路径目前只有 procedure 的例子，
  domain 候选标签与已确认标签的展示细节使用频率也偏低。这些在主文档里已设计完整，属于"模型完备但
  场景稀薄"，不必删除定义，但近期实现优先级排在核心能力之后。
- **旅程验证过程中新发现、需要补充设计的缺口**：domain 和 flow 都缺少像 `procedure`/`project` 那样
  明确的目录结构和编译产物文件名，domain 还缺专属 MCP 方法。这不是"K3 层该不该编译"的能力问题
  （K3 本就应该像 K2 一样编译），只是主文档没把这层规格写全，建议下一轮直接补到 6.2/6.3 节和 8 节，
  不需要再等更多旅程验证。同一类问题也出现在 `project` 上：把场景从"项目交接"抽象成"项目管理"
  后才发现"只读进度查询"（旅程 H）和"接手交接"（旅程 C）被主文档隐含地合并成一件事，实际应该是
  `get_project_context` 下的两种意图，读操作不该触发任何交接协议（如强制刷新 `session_anchor`）。
  旅程 I 还暴露一个更根本的交互模式缺口：主文档 7.4 节和 8 节的 MCP 方法全部假设"先有一个自然语言
  问题，再返回结果"，没有设计"编码时检测到代码依赖变化，主动推送对应 `shared_capability` 用法"这
  种由信号（依赖/import）而不是由提问触发的加载方式，这个缺口比前面几项都更基础，建议优先评估。

## 5. 后续建议

- 每新增一个用户故事，先按第 1.1 节的标准判断"这是不是一个新角色"（换个人做会不会本质不同），
  再决定是新增角色还是归入某个角色下的新场景。角色数量应保持很少（目前 3 个），场景可以随需求
  自由增长，不会让角色列表本身也变得大而全。
- 主文档 7.3 节的"场景分类与知识加载策略"可以逐步替换引用方式：新增能力前先在本文档补一条用户
  故事和一条旅程，再回到主文档补检索配方，保证"先有场景，再有结构"的顺序在后续迭代中保持一致。
- 第 4 节里标记为"暂缓投入"的部分不代表设计错误，只代表当前证据不足；"规格缺口"的部分则应该
  尽快补齐，因为能力本身已经被旅程验证是必需的，只是文档没写全，不需要等更多证据。
- 命名场景时先做一次"再抽象一层"的检查：如果场景名字只描述了一个具体触发时刻（比如"交接"、
  "申请"），先问一句"这个触发时刻背后，是不是还有同一类任务的其他触发方式没被看到"。项目管理
  场景就是这样从"项目交接场景"抽象出来的，抽象后立刻多出一个之前没写的子场景（US-15/旅程 H）；
  下次遇到类似"只写了一个动作名"的场景，应该先做这一步，而不是等用户指出来才补。
